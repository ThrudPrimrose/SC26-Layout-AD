# E6 / access_analysis

LOKI-driven analysis of ICON's `velocity_tendencies` subroutine. Produces
the three inputs the rest of the E6 pipeline needs:

1. **25 loop nests + their role-tagged access patterns** ‚Üí
   `velocity_advection_preprocessed.analysis.md`
2. **The 6 representative loop nests** (one per pattern class ‚Äî manually
   pinned + auto-tiebroken) ‚Üí `layout_candidates.json` + `chosen_loopnests.md`
3. **The 6 canonical array groups** (cv / ch / f / s / n / lm) ‚Üí
   `canonical_array_groups.json`

A small companion reporter, [`find_groups_and_loopnests.py`](find_groups_and_loopnests.py),
prints (1)+(2) side-by-side and writes a single-file summary at
`groups_and_loopnests.json` for downstream tools that want the full
list without re-parsing the upstream JSONs.

## Files

| File | Kind | Purpose |
|---|---|---|
| `analyze_access.py` | script | LOKI entry. Parses `velocity_advection_preprocessed.f90` ‚Üí `*.analysis.md`. |
| `select_loopnests.py` | script | Picks 6 representative nests by pattern class + tiebreaker (loopnest_1 manually pinned). ‚Üí `layout_candidates.json` + `chosen_loopnests.md`. |
| `generate_canonical_groups.py` | script | Derives the 6-group layout taxonomy (cv / ch / f / s / n / lm) from `layout_candidates.json`. Connectivity aliases are the one piece LOKI can't expose ‚Äî they're hardcoded in this script. |
| `find_groups_and_loopnests.py` | script | Reporter. Prints the 6 access groups + the 6 chosen loopnests side-by-side and writes `groups_and_loopnests.json` (read-only summary). |
| `velocity_advection_preprocessed.f90` | input | Fortran source analyzed. Preprocessed so `#include` / `!$OMP` are already expanded. |
| `velocity_advection_preprocessed.analysis.md` | output | Auto-generated by `analyze_access.py`. 25 nests + array groups + per-array summaries. |
| `layout_candidates.json` | output | Structured record of the 6 chosen nests and candidates per class. Consumed by `../generate_winners.py` and `../build_layout_crossproduct.py`. |
| `chosen_loopnests.md` | output | Human-readable mirror of `layout_candidates.json`. |
| `canonical_array_groups.json` | output | Auto-generated 6-group taxonomy. Every group is also tagged with its storage axis (`IC` for cv/ch/f/s/lm; `IN` for n). Consumed by `../generate_winners.py` and `../build_layout_crossproduct.py`. |
| `groups_and_loopnests.json` | output | Auto-generated by `find_groups_and_loopnests.py`. Single-file mirror of the 6 groups + 6 loopnests for downstream use. |

## Running

```bash
source ../../common/activate.sh   # venv with LOKI installed

python3 analyze_access.py velocity_advection_preprocessed.f90
# -> velocity_advection_preprocessed.analysis.md

python3 select_loopnests.py
# -> layout_candidates.json, chosen_loopnests.md

python3 generate_canonical_groups.py
# -> canonical_array_groups.json

python3 find_groups_and_loopnests.py
# -> groups_and_loopnests.json (and a printed summary)
```

## How the 6 canonical groups are derived

`generate_canonical_groups.py` walks every (array, touching-nest-set)
pair in `layout_candidates.json` and assigns it one of six tags by
rule:

| Group | Rule |
|---|---|
| `n` ‚Äî CONN | Name matches `p_patch%{cells\|edges\|verts}%(edge\|cell\|vertex\|neighbor\|quad)_(idx\|blk)`, OR is one of the local aliases `icidx / icblk / ieidx / ieblk / ividx / ivblk / iqidx / iqblk / incidx / incblk`. |
| `f` ‚Äî FIELDS | Name begins with `p_prog%` or `p_diag%`. |
| `s` ‚Äî STENCIL | Name begins with `p_metrics%`, `p_int%`, or `p_patch%{edges,cells}%*` (non-connectivity). |
| `cv` ‚Äî COMPUTE_VERT | Work array (`z_*`, `cfl_*`, `lev*mask`) that appears in at least one nest with shape `v.h` AND `full_vert` range. |
| `ch` ‚Äî COMPUTE_HORIZ | Work array that doesn't meet the `cv` criterion (boundary sweeps + partial-vertical stencils). |
| `lm` ‚Äî LEVMASK | Level-mask 2D arrays (`levmask`, `levelmask`) ‚Äî split out from `cv` so the `(nblks, nlev) ‚Üí (nlev, nblks)` transpose can be expressed independently of the rest of `cv`. |

Everything except the connectivity aliases is derived from LOKI output.
The alias list is hardcoded because `analyze_access.py` filters out
indirection-array accesses (`INDIRECT_ARRAYS` in its name set) so those
arrays never appear in any nest's `arrays` field ‚Äî but they're still
swept as their own axis in the real layout search, so the `n` group
must be emitted.

## Each group is tagged with its storage axis

Every group carries a `group_axis` entry in the JSON so the downstream
cross-product builder knows which storage axis the group consumes:

| Group | Axis | Why |
|---|---|---|
| `cv`, `ch`, `f`, `s`, `lm` | `IC` | (h,v,b) arrays ‚Äî choice is whether `je` or `jk` is innermost: `h_first` vs `v_first` |
| `n` | `IN` | (N,2) connectivity ‚Äî choice is SoA vs AoS |

The builder (`../build_layout_crossproduct.py`) uses this tag to project
each V-winner onto the right axis before voting. V-ids that differ only
in the opposite axis (e.g. V3 vs V4 on the IC axis, or V1 vs V3 on the
IN axis) collapse to the same candidate, which keeps the final sweep
small and free of aliased configs.

**Caveat.** The `cv` / `ch` boundary in the real sweep
(`icon-artifacts/velocity/sc26_layout/permute_stage8.py`) is drawn by
the *write direction* of each producer loop (jk-outer write ‚Üí cv;
je-inner write ‚Üí ch). LOKI's current output doesn't split reads from
writes, so this script falls back to "touching nest shape". Expect
small differences (e.g. `z_v_grad_w` lands in `cv` here but in `ch` in
the reference repo). If exact parity is needed, extend
`analyze_access.py` to emit per-array write-site metadata and update
the heuristic accordingly.
